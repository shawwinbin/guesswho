# 历史人物猜谜无限关卡设计方案

> 为现有“微信小游戏 + 轻国风卡通”版本增加连续闯关能力，形成“每局一关、越往后越难、失败后可从最近解锁关继续”的长期留存主线。

## 概述

### 目标

在不改变核心玩法的前提下，为当前项目增加一套稳定、可持久化、可逐步扩展的无限关卡系统。

这一轮的重点不是做大地图冒险，而是让现有首页、游戏页、结果页形成清晰的闯关主线：

- 首页有可横滑的关卡带
- 游戏页明确显示当前挑战的是第几关
- 结果页能给出“通关、解锁、继续挑战”的反馈
- 难度随关卡提升，主轴为人物池变难，而不是减少提问次数

### 非目标

这一版不包含以下内容：

- 云端同步或账号级关卡存档
- 排行榜、赛季、每日挑战
- 大地图章节系统
- 道具、复活、跳关、扫荡
- 复杂数值养成

## 产品原则

### 1. 每局一关

每一次创建游戏会话都对应一个明确关卡，用户感知为“正在挑战第 N 关”。

### 2. 难度来自人物池

提问次数保持基本稳定，难度提升主要由人物认知门槛递增实现。

### 3. 失败不掉档

失败后不回到第 1 关，保留最近已解锁关，降低挫败感。

### 4. 首页就能看到进度

用户一进入首页就能看到自己打到哪里、下一关在哪、哪些关已通关。

### 5. 状态边界清晰

“单局会话状态”和“闯关生涯状态”分离，避免后续功能扩展时状态耦合。

## 体验设计

### 首页

首页在保留现有 Stitch 化主视觉的基础上，新增一条可横滑关卡带，放在主视觉和挑战卡之间。

### 关卡带结构

每个关卡节点是一张小关卡牌，包含：

- `第 N 关`
- 难度称号
- 状态标记

状态分为三类：

- 已通关：朱红印章、已完成感更强
- 当前解锁：金边高亮、轻呼吸感、作为当前挑战入口
- 未解锁：降饱和、小锁图标、不可点击

### 首页主按钮

首页主按钮从“开始游戏”调整为：

- `继续挑战 第 N 关`

按钮上方补一行进度摘要：

- `已通关 X 关`
- `最高解锁 第 Y 关`

### 交互

- 关卡带默认滚动到 `currentLevel` 附近
- 用户可以点击任意已解锁关切换当前挑战关
- 用户不能点击未解锁关

### 游戏页

游戏页保持现有“关卡/HUD”风格，但新增明确的关卡信息条。

### 顶部关卡信息

在标题区域增加：

- `LEVEL N`
- 难度称号
- 轻量说明，如 `人物更冷门` / `人物更易混淆`

### 本关目标提示

在问答卷轴或状态条附近增加一条本关目标说明：

- `通关条件：在 20 次提问内猜中本关人物`

### 反馈预期

在操作台区域增加一条轻提示：

- `胜利后解锁第 N+1 关`

这样用户在游戏过程中始终知道：

- 自己正在打哪一关
- 这一关为什么更难
- 成功后会得到什么推进

### 结果页

结果页负责承接闯关推进感。

### 胜利态

- 标题区增加 `第 N 关通关`
- 奖励区后新增 `新关解锁卡`
- 主按钮改成 `挑战第 N+1 关`

### 失败态

- 明确提示 `已为你保留至第 N 关`
- 主按钮改成 `从第 N 关继续`
- 次按钮仍为 `返回首页`

### 结果页关卡带

结果页底部增加一段较短的关卡带，仅展示当前关附近几个节点，用于强化“你刚刚打到这里”的战报感。

## 关卡模型

### 核心字段

新增一份独立的本地持久化关卡进度对象：

```ts
interface LevelProgress {
  currentLevel: number
  highestUnlockedLevel: number
  highestClearedLevel: number
  levelStreak: number
  lastResult: 'win' | 'lose' | null
}
```

字段定义：

- `currentLevel`：当前准备挑战的关卡
- `highestUnlockedLevel`：最高已解锁关
- `highestClearedLevel`：最高已通关关
- `levelStreak`：连续通关数，先存储，后续可用于奖励感
- `lastResult`：上一局结果，便于首页/结果页显示刚刚解锁的反馈

### 默认值

首次进入时：

```ts
{
  currentLevel: 1,
  highestUnlockedLevel: 1,
  highestClearedLevel: 0,
  levelStreak: 0,
  lastResult: null,
}
```

### 存储策略

- 关卡进度单独存储，不与游戏设置混存
- 清除本地数据时一并清理
- 旧用户没有关卡进度时，平滑迁移到默认值

## 胜负与切关规则

### 胜利

- 当前关记为已通关
- 解锁下一关
- `highestClearedLevel = max(highestClearedLevel, currentLevel)`
- `highestUnlockedLevel = max(highestUnlockedLevel, currentLevel + 1)`
- `currentLevel = currentLevel + 1`
- `levelStreak += 1`
- `lastResult = 'win'`

### 失败

- 不回退已解锁进度
- `currentLevel` 保持在当前已解锁最高关
- `highestUnlockedLevel` 不变
- `highestClearedLevel` 不变
- `levelStreak = 0`
- `lastResult = 'lose'`

### 手动切关

- 首页允许点击已解锁关卡，切换 `currentLevel`
- 未解锁关卡不可进入
- 游戏页“重新开始”只重开当前关，不影响闯关生涯

## 难度系统

### 核心原则

难度递增主轴是人物池难度，不是提问次数。

因此需要把关卡等级作为建局参数传给后端，让后端按关卡选择更难的人物。

### 数据策略

为人物数据增加显式 `difficulty` 字段，推荐范围为 `1-5`：

- `1`：大众高认知人物
- `2`：常见但不算第一反应的人物
- `3`：需要中等推理量的人物
- `4`：相对冷门或容易混淆的人物
- `5`：高难人物，要求用户提出更精准的问题

优先采用人工标注，而不是纯算法猜测。原因是“难度”本质是产品定义，不只是数据特征。

### 关卡到难度映射

推荐采用平滑分段：

- `1-3关`：以 `difficulty 1` 为主
- `4-6关`：以 `difficulty 2` 为主
- `7-10关`：以 `difficulty 3` 为主
- `11-15关`：以 `difficulty 4` 为主
- `16关+`：以 `difficulty 5` 为主，可少量混入 `4`

这样可以兼顾：

- 新手前期连续过关
- 中段开始明显感知“人物更难”
- 后期进入稳定高难挑战区

## 技术架构

### 前端职责

前端负责：

- 持久化闯关生涯状态
- 展示关卡带和关卡文案
- 创建会话时传入当前关卡
- 在结果页根据胜负更新进度

前端不负责：

- 自行抽人物
- 自行决定本关人物难度

### 后端职责

后端负责：

- 接收 `level`
- 根据 `figureScope + level` 映射到目标难度池
- 从合适的人物集合中抽取本局人物
- 将当前关信息写入会话快照，便于前端展示和恢复

### 状态边界

#### 单局会话状态

继续由 `useGameSession` 管理：

- sessionId
- phase
- history
- guesses
- isWinner
- revealedName
- remainingQuestions

#### 闯关生涯状态

新增独立状态模块管理：

- currentLevel
- highestUnlockedLevel
- highestClearedLevel
- lastResult

两者必须分离，不能塞进一个大对象中混用。

## 数据流

### 首页进入

1. 读取本地 `LevelProgress`
2. 关卡带滚动到 `currentLevel`
3. 主按钮显示 `继续挑战 第 currentLevel 关`

### 开始挑战

1. 点击首页主按钮
2. 进入游戏页
3. 建局请求携带：
   - `questionLimit`
   - `figureScope`
   - `level`

### 游戏结束

1. 游戏页根据结果跳转结果页
2. 结果页读取当前关卡与胜负结果
3. 更新 `LevelProgress`
4. 根据新状态决定按钮文案与下一步去向

## 错误处理

- 若本地关卡数据损坏，自动回退默认值
- 若后端未返回有效关卡信息，前端回退为当前本地关卡展示
- 若某一难度池无人物可用，后端回退到最近可用难度池
- 若旧会话恢复时缺少关卡字段，按当前本地 `currentLevel` 兜底

## 测试策略

### 前端测试点

- 首次进入时初始化为第 `1` 关
- 首页主按钮、关卡带能正确反映 `currentLevel`
- 首页点击已解锁旧关可切换，未解锁关不可切换
- 胜利后正确推进下一关
- 失败后不丢失已解锁进度
- 清除本地数据后关卡进度恢复默认值
- 老用户无关卡进度数据时可平滑迁移

### 后端测试点

- `createSession` 可接收 `level`
- 不同 `level` 会映射到不同难度人物池
- 当目标难度池为空时能正确回退
- 会话快照能返回关卡相关字段

### 页面验收点

- 首页存在可横滑关卡带，并能看到已通关、当前解锁、未解锁三种状态
- 游戏页能明确看到当前 `LEVEL N`
- 结果页在胜利/失败两种分支下都能给出正确的推进反馈
- 用户从结果页返回首页后，首页进度与结果一致

## 实施顺序建议

1. 增加关卡进度本地存储与工具函数
2. 为前后端建局协议增加 `level`
3. 为人物数据补充 `difficulty`
4. 在后端实现按关卡抽人
5. 改首页关卡带与主按钮
6. 改游戏页关卡信息条
7. 改结果页推进反馈
8. 补充前后端测试与回归

## 成功标准

满足以下条件即可视为本次设计落地成功：

- 用户能清晰感知自己在“连续闯关”
- 难度上升主要来自人物池，而不是次数惩罚
- 胜利能推进，失败不掉档
- 首页、游戏页、结果页三页形成统一的关卡叙事
- 关卡状态与单局状态边界清晰，方便后续继续扩展
